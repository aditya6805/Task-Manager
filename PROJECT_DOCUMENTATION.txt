PROJECT TITLE

Team Task Manager

============================================================

1. PROJECT OVERVIEW

Team Task Manager is a modern full-stack web application designed to help teams plan work, assign responsibilities, track progress, review submissions, and keep projects organized in one place. It combines a React-based frontend, a Node.js and Express backend, Firebase authentication, and MongoDB data storage to create a production-style task and project management experience.

The application is built for collaborative work environments where multiple users need clear visibility into what needs to be done, who is responsible for it, and what stage each task is currently in. It is especially useful for small teams, startup environments, student groups, internal department workflows, and portfolio-style project coordination.

The project has been developed in a way that is easy to understand for beginners, while still reflecting the structure and engineering standards expected in a professional full-stack application.

============================================================

2. PROBLEM STATEMENT

Many teams still manage tasks using scattered tools, chat messages, spreadsheets, or disconnected note-taking apps. This creates confusion, missed deadlines, duplicate work, and a lack of visibility into who is doing what. Team members often do not know which tasks are pending, which are in progress, which are overdue, and which tasks have already been completed.

Another common problem is that many basic task systems do not support proper role-based access. In real team environments, some users should be allowed to create and manage projects, while others should only be able to view assigned work and update progress. Without authentication and access control, team coordination becomes unsafe and difficult to trust.

Team Task Manager addresses these issues by providing a single centralized application where projects, tasks, users, roles, and activity tracking are all connected in a structured way.

============================================================

3. PURPOSE OF THE PROJECT

The purpose of Team Task Manager is to provide a practical digital workspace for team collaboration. It helps teams plan work more effectively, monitor progress in real time, and reduce communication gaps between administrators and team members.

The project also serves as a strong demonstration of modern full-stack development practices. It shows how a frontend application can communicate securely with a backend API, how Firebase authentication can be combined with MongoDB user records, and how task and project data can be organized for an efficient dashboard experience.

7. COMPLETE WORKFLOW OF THE APPLICATION

The application follows a clear end-to-end workflow:

1. A new user signs up or logs in using Firebase authentication.
2. After authentication, the frontend stores the Firebase ID token locally.
3. The user profile is synchronized with the backend MongoDB database.
4. The React application loads protected pages through authenticated routes.
5. The frontend sends API requests to the backend using Axios.
6. The backend verifies the Firebase token through middleware.
7. The backend checks the user role and access permissions.
8. The backend reads or updates project and task data in MongoDB.
9. The backend returns structured JSON responses.
10. The frontend updates the dashboard, task list, or project view based on the returned data.

This flow ensures that the frontend is responsible for the user interface and experience, while the backend is responsible for secure business logic, authorization, and data handling.

============================================================

8. STEP-BY-STEP USER JOURNEY

A typical user journey in the application looks like this:

1. The user opens the app in the browser.
2. They reach the Home page and choose to log in or sign up.
3. Firebase authenticates their credentials.
4. The system stores the token and creates or syncs the user profile in the backend.
5. The user is redirected into the protected application area.
6. The dashboard displays their task and project summary.
7. They navigate to tasks or projects depending on their responsibilities.
8. They view details, update status, add submissions, or review feedback.
9. If they are an Admin, they can create projects, create tasks, assign members, and manage team work.
10. When they log out, the token is removed and access to protected routes is revoked.

This journey is intentionally simple so that even a beginner can understand how the application is used.

9. FRONTEND TECHNOLOGIES USED

The frontend is built with the following technologies:

- React 18: Used for building the user interface with reusable components and reactive state updates.
- Vite: Used as the build tool and development server because it offers fast startup and fast hot reloading.
- React Router DOM: Used to manage page navigation without full page reloads.
- Tailwind CSS: Used for styling through utility classes, which makes the UI fast to build and easy to maintain.
- Axios: Used for communication with the backend API.
- Firebase Web SDK: Used for login, signup, authentication state management, and token generation.

Why these technologies were used:

React gives the project a component-based structure. Vite improves developer experience during build and development. Tailwind CSS keeps the UI clean and responsive without requiring a large custom CSS architecture. Axios simplifies API calls. Firebase makes authentication reliable and secure without building a custom auth system from scratch.

============================================================

10. BACKEND TECHNOLOGIES USED

The backend uses the following technologies:

- Node.js: Provides the JavaScript runtime for server-side execution.
- Express.js: Used to create the REST API and manage routes.
- Mongoose: Used to interact with MongoDB in a structured way.
- MongoDB: Stores application data such as users, projects, and tasks.
- Firebase Admin SDK: Used to verify Firebase authentication tokens on the server.
- CORS: Used to safely control which frontend origins can access the API.
- dotenv: Used to load environment variables.
- express-validator: Included as a backend dependency for request validation support.
- nodemon: Used in development so the server reloads automatically when files change.

Why these technologies were used:

Node.js and Express are well suited for REST APIs. MongoDB is flexible for project and task records that can evolve over time. Mongoose helps keep data access organized. Firebase Admin SDK allows the backend to trust authentication tokens issued by Firebase. CORS and dotenv support secure deployment and environment configuration.

============================================================

11. DATABASE USED

The primary database technology is MongoDB.

MongoDB stores application records in a document-based format. This is useful for a task management system because projects and tasks can contain flexible fields such as members, assigned users, feedback, submission history, and status timelines.

The project uses Mongoose models on the backend to interact with the database in a structured way. The codebase also includes a fallback in-memory model store for local development when a real MongoDB connection string is missing or still using a placeholder value. This makes the application easier to work with during setup and demonstration.

============================================================

12. AUTHENTICATION & SECURITY

Authentication is handled through Firebase Authentication. Users sign in using Firebase, and Firebase returns an ID token that represents the authenticated session.

The frontend stores this token locally and sends it with every API request in the Authorization header. The backend uses middleware to verify the Firebase token before allowing access to protected routes.

Security features include:

- Protected routes on the frontend.
- Token-based API access.
- Server-side authentication verification.
- Role-based authorization checks.
- Route-level access control for admin-only actions.
- CORS origin restrictions so only allowed frontend domains can communicate with the API.

This design is secure because the backend never trusts the frontend alone. It validates every authenticated request before returning sensitive data or allowing data changes.

============================================================

13. API STRUCTURE AND COMMUNICATION

The application uses a REST-style API structure.

Main API areas:

- /api/health
- /api/tasks
- /api/projects
- /api/dashboard
- /api/users

Frontend to backend communication flow:

1. The React UI triggers an action such as loading tasks or creating a project.
2. A service function sends the request through the shared Axios client.
3. The Axios client automatically attaches the Firebase token from local storage.
4. The backend authentication middleware verifies the token.
5. The route handler forwards the request to the appropriate controller.
6. The controller reads or writes data in MongoDB.
7. The backend sends a JSON response with status, message, and data.
8. The frontend updates the interface using the response.

This structure keeps the app modular, predictable, and easy to maintain.

14. FOLDER STRUCTURE EXPLANATION

Frontend folder structure:

- client/src/components: Reusable UI building blocks such as cards, forms, tables, dialogs, and navigation elements.
- client/src/context: Global state management for authentication and application-level data.
- client/src/layouts: Shared layout shells used across internal pages.
- client/src/pages: Full screens such as Home, Login, Signup, Dashboard, Tasks, Projects, and detail pages.
- client/src/services: API communication helpers and Firebase-related logic.

Backend folder structure:

- server/config: Database and Firebase initialization logic.
- server/controllers: Business logic for tasks, projects, dashboard data, and users.
- server/middleware: Authentication and role-based access control.
- server/models: Data models and model exports.
- server/routes: API route definitions.
- server/utils: Shared helper utilities for response handling.

This structure separates concerns clearly, which is one of the most important habits in professional full-stack development.

15. CORE MODULES EXPLANATION

The most important modules in the project are:

- AuthContext: Manages authentication state, current user data, loading state, role detection, and logout behavior.
- firebaseService: Handles Firebase login, signup, logout, auth listeners, and token retrieval.
- api.js: Creates the reusable Axios client and injects authentication tokens into API requests.
- userService: Synchronizes Firebase-authenticated users with the MongoDB backend and handles profile-related requests.
- ProtectedRoute: Prevents unauthenticated users from entering protected sections of the app.
- MainLayout: Provides the common authenticated app structure.
- Dashboard: Displays summarized project and task information.
- TaskForm and ProjectForm: Support creation and editing flows.
- TaskTable: Presents task data in a structured view.
- ConfirmDialog: Provides safe confirmation before destructive actions.

These modules form the functional backbone of the application.

19. FIREBASE INTEGRATION

Firebase is used for the authentication layer of the application.

It handles:

- User signup.
- User login.
- Login state tracking.
- Token generation.
- Sign out.
- Auth state changes.

The frontend listens to auth changes and keeps the application state updated. When a user authenticates, the Firebase token is stored and then attached to API requests.

Firebase was used because it saves development time, provides a trusted authentication system, and removes the need to build a custom password-handling flow. This improves both security and reliability.

============================================================

20. MONGODB INTEGRATION

MongoDB is used as the data store for application records.

The backend uses Mongoose models to manage:

- Users
- Projects
- Tasks

MongoDB is a strong fit here because the data is naturally document-oriented. For example, a task can include assignees, submissions, feedback entries, status information, and project references.

The backend is also designed to handle local development gracefully. If the MongoDB URI is missing or still a placeholder, the application can fall back to an in-memory store. This helps during setup and testing.

21. DEPLOYMENT DETAILS

The application is designed for a split deployment model.

Frontend deployment: Vercel

- The frontend is suitable for Vercel because it is a Vite-based React application.
- The build output is generated as a static production bundle.
- Vercel serves the frontend efficiently and supports environment variables for the client-side build.
- The client-side API URL should point to the deployed backend.

Backend deployment: Render

- The backend is suitable for Render because it is a Node.js and Express server.
- Render can run the API as a web service.
- The backend exposes protected REST endpoints that the Vercel frontend consumes.
- The backend environment must include the database and Firebase credentials.

Simple deployment architecture:

1. Vercel hosts the React frontend.
2. Render hosts the Express API.
3. The frontend sends requests to the Render API URL.
4. The backend verifies auth tokens, reads MongoDB data, and returns JSON.

This is a clean and professional architecture for a modern full-stack project.

22. ENVIRONMENT VARIABLES EXPLANATION

Client-side environment variables:

- VITE_API_URL: Base URL for the backend API.
- VITE_DEV_API_PROXY_TARGET: Optional local proxy target during development.
- VITE_FIREBASE_API_KEY: Firebase web API key.
- VITE_FIREBASE_AUTH_DOMAIN: Firebase auth domain.
- VITE_FIREBASE_PROJECT_ID: Firebase project ID.
- VITE_FIREBASE_STORAGE_BUCKET: Firebase storage bucket.
- VITE_FIREBASE_MESSAGING_SENDER_ID: Firebase messaging sender ID.
- VITE_FIREBASE_APP_ID: Firebase application ID.

Server-side environment variables used by the backend code:

- PORT: Port on which the Express server runs.
- NODE_ENV: Environment mode such as development or production.
- MONGODB_URI: MongoDB connection string.
- CORS_ORIGIN: Allowed frontend origins for API access.
- FIREBASE_PROJECT_ID: Firebase project identifier for admin initialization.
- FIREBASE_PRIVATE_KEY: Firebase private key used by the admin SDK.
- FIREBASE_CLIENT_EMAIL: Firebase service account email used by the admin SDK.

These variables keep configuration outside the source code, which is a standard professional practice.

25. ERROR HANDLING STRATEGY

The application uses layered error handling.

Frontend error handling:

- API calls are wrapped in try/catch blocks.
- User-friendly error messages are shown when requests fail.
- Loading states are displayed while data is being fetched.
- Empty states are shown when no data is available.

Backend error handling:

- Controllers return structured JSON responses.
- Validation errors are returned with clear messages.
- Unauthorized or forbidden actions are blocked by middleware and role checks.
- The server includes a 404 route for unknown API paths.
- Global handlers capture uncaught exceptions and unhandled rejections.

This strategy helps the application fail safely and makes debugging much easier.
